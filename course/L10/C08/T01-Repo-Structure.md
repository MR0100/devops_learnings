# L10/C08/T01 — Repository Structure

## Learning Objectives

- Structure Terraform repo
- Apply per scale

## Patterns

Several patterns; depend on:
- Team size
- Number of environments
- Number of accounts
- Compliance needs

## Small Team

```
infrastructure/
├── .github/workflows/
├── modules/
│   ├── vpc/
│   └── eks/
├── envs/
│   ├── dev/
│   ├── staging/
│   └── prod/
└── README.md
```

Single repo; all infrastructure.

## Medium

```
infrastructure/
├── .github/workflows/
├── modules/
├── live/
│   ├── shared/                 # org-level
│   ├── prod/
│   │   ├── us-east-1/
│   │   │   ├── network/
│   │   │   ├── apps/
│   │   │   └── data/
│   │   └── eu-west-1/
│   ├── staging/
│   └── dev/
└── README.md
```

Per region; per layer.

## Large Org

Multiple repos:
- `infrastructure-modules/` (shared modules)
- `infrastructure-shared/` (org / shared services)
- `infrastructure-team-a/` (team A workloads)
- `infrastructure-team-b/`
- ...

Per-team independence.

## Mono Repo

Pros:
- Single source of truth
- Coordinated changes
- Easier visibility

Cons:
- Permissions hard to scope
- Scaling issues
- All-or-nothing CI

## Multi Repo

Pros:
- Per-team independence
- Permissions per repo
- Clear ownership

Cons:
- Coordinated changes harder
- Shared modules versioning

## Modules Repo

```
infrastructure-modules/
├── README.md
├── CHANGELOG.md
├── vpc/
│   ├── main.tf
│   ├── variables.tf
│   ├── outputs.tf
│   ├── README.md
│   └── examples/
├── eks/
├── rds/
└── ...
```

Tagged releases (v1.2.3).

Consumed:
```hcl
module "vpc" {
  source  = "git::https://github.com/me/infrastructure-modules.git//vpc?ref=v1.2.3"
}
```

## Per-Workload State

Each leaf directory = own state:
```
live/prod/us-east-1/vpc/      # state for VPC
live/prod/us-east-1/eks/      # state for EKS
live/prod/us-east-1/rds/      # state for RDS
```

Independent applies; clear ownership.

## State Backend per Env

```hcl
# backend.tf in prod/
backend "s3" {
  bucket = "prod-tfstate"
  key    = "..."
}

# backend.tf in dev/
backend "s3" {
  bucket = "dev-tfstate"
  key    = "..."
}
```

Different IAM; different blast radius.

## File Naming

Convention:
- `main.tf`: primary resources
- `variables.tf`: vars
- `outputs.tf`: outputs
- `versions.tf`: provider + Terraform version constraints
- `providers.tf`: provider configs
- `locals.tf`: locals
- `data.tf`: data sources

Optional:
- `iam.tf`, `network.tf`, etc. for organization

## .gitignore

```
.terraform/
.terraform.lock.hcl  # decide: commit or not
*.tfstate
*.tfstate.backup
*.tfvars             # if contains secrets
.terraform.tfstate.lock.info
```

Commit lock.hcl: usually yes (deterministic).

## CI Workflows

```yaml
# .github/workflows/terraform.yaml
on:
  pull_request:
    paths: ['live/**', 'modules/**']

jobs:
  validate:
    steps:
    - run: terraform fmt -check
    - run: terraform validate
    - run: tflint
    - run: tfsec
    - run: checkov -d .
  
  plan:
    strategy:
      matrix:
        dir: ['live/dev/us-east-1/vpc', 'live/dev/us-east-1/eks']
    steps:
    - run: cd ${{ matrix.dir }} && terraform plan
```

Per-directory plans.

## CODEOWNERS

```
# CODEOWNERS

modules/eks/        @platform-team
live/prod/**        @platform-team @security-team
live/dev/**         @platform-team
```

Auto-assign reviewers.

## Branching Strategy

- main → production
- Feature branches → PRs
- Atlantis or CI plans
- Merge after approval → apply

## Tagging Releases (Modules)

```bash
git tag -a v1.2.3 -m "Release 1.2.3"
git push origin v1.2.3
```

Consumers pin.

## Versioning Lock

`.terraform.lock.hcl`:
- Tracks provider versions + hashes
- Commit to repo
- Ensures team uses same providers

```bash
terraform init -upgrade   # update lock
```

## Variables

Per env:
```
live/prod/us-east-1/vpc/
├── main.tf
├── variables.tf
├── terraform.tfvars       # prod-specific
└── backend.tf
```

Or via Terragrunt for DRY.

## Documentation

Beyond per-module READMEs:
- Architecture diagrams (Mermaid in README)
- Runbooks (separate / docs/)
- ADRs (architectural decisions)
- Onboarding guide

## Security

For secrets in repo:
- Don't commit .tfvars with secrets
- Use SOPS / Vault / AWS Secrets Manager
- .gitignore enforces
- Pre-commit hooks (detect-secrets, trufflehog)

## Pre-Commit Hooks

```yaml
# .pre-commit-config.yaml
repos:
- repo: https://github.com/antonbabenko/pre-commit-terraform
  rev: v1.81.0
  hooks:
  - id: terraform_fmt
  - id: terraform_validate
  - id: terraform_tflint
  - id: terraform_tfsec
  - id: terraform_docs
```

Run on commit:
```bash
pre-commit install
```

## Tagging Resources

Standard tags via provider:
```hcl
provider "aws" {
  default_tags {
    tags = local.common_tags
  }
}

locals {
  common_tags = {
    Project    = "my-project"
    Env        = var.env
    Team       = var.team
    ManagedBy  = "terraform"
    Repo       = "github.com/me/infrastructure"
  }
}
```

## Migration Strategy

Adopting structure incrementally:
1. New work in new structure
2. Move existing piece by piece
3. Use `moved` blocks
4. Test plans = no changes
5. Refactor over months

## Common Mistakes

- Flat structure (1 dir for all)
- Mixed envs in one state
- Modules in main repo (consumer pulls everything)
- No CODEOWNERS
- Secrets committed

## Best Practices

- Modules + envs separated
- Per-leaf state
- Per-env backends
- CODEOWNERS
- Pre-commit hooks
- terraform-docs
- CI validates
- Atlantis or TFC for PRs

## Example Real Org

```
my-org/
├── infrastructure-modules/     # versioned modules
├── infrastructure-shared/       # org-level (CloudTrail, etc.)
├── infrastructure-platform/    # platform team workloads
└── infrastructure-team-X/      # team workloads
    ├── live/
    │   ├── prod/...
    │   └── dev/...
    └── modules/                  # team-specific (private)
```

Separate repos; clear ownership.
