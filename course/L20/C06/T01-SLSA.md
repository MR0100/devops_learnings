# L20/C06/T01 — SLSA Framework

## Learning Objectives

- Understand SLSA
- Apply levels

## SLSA

Supply-chain Levels for Software Artifacts:
- Google initiative
- Framework for build security
- Levels of assurance

## Levels

### L1
- Documented build
- Provenance available

### L2
- Hosted build platform
- Tamper-resistant provenance

### L3
- Hardened builds (isolated)
- Authenticated provenance

### L4
- Two-person review
- Hermetic + reproducible builds

For: progressive maturity.

## Provenance

Build metadata:
- Source (commit hash)
- Builder
- Materials (inputs)
- Outputs

For: traceability.

## Threats Addressed

- Source compromise
- Build compromise
- Distribution tampering
- Dependency confusion

## Level 1

```yaml
provenance:
  builder: github-actions
  buildType: workflow
  invocation:
    configSource: ...
```

Document; not signed.

For: starting point.

## Level 2

Provenance:
- Signed
- Generated by build platform
- Not tampered

Tools: GitHub Actions, GitLab CI.

## Level 3

Build isolation:
- Ephemeral environment
- No long-lived secrets
- No interference

For: stronger.

## Level 4 (Rare)

Hermetic:
- Inputs explicit
- Reproducible
- Two-person review

For: highest. Critical infra.

## GitHub Provenance

```yaml
- uses: actions/attest-build-provenance@v1
  with:
    subject-path: 'dist/*'
```

Generates + publishes to Rekor.

## Verify

```bash
gh attestation verify dist/* --owner myorg
```

## Cosign + SLSA

```bash
cosign verify-attestation \
  --certificate-identity ... \
  --certificate-oidc-issuer ... \
  --type slsaprovenance \
  IMAGE
```

## SBOM + Provenance

Both:
- SBOM: what's in artifact
- Provenance: how built

Together: full picture.

## Why SLSA

- Detect tampering
- Audit trail
- Compliance (EU CRA, US EO)

## Threat Models

### Source
Attacker commits malicious code.
SLSA: signed commits, two-person review.

### Build
Attacker compromises CI.
SLSA: isolated builders, attestation.

### Dependency
Attacker compromises dep.
SLSA: SBOM + verify.

### Distribution
Attacker swaps artifact.
SLSA: signed + provenance verified.

## Maturity Adoption

```
L0: nothing (typical default)
L1: simple add (some)
L2: GitHub / GitLab provides (many)
L3: harder (some)
L4: rare
```

## Compliance

US Executive Order 14028:
- SBOM
- Attestations
- For federal vendors

For: increasingly required.

## EU Cyber Resilience Act

Similar mandates for EU.

## Verification Policy

K8s admission:
```yaml
- verifyImages:
  - attestors:
    - entries:
      - keyless:
          subject: "https://github.com/myorg/myrepo/.github/workflows/build.yml@refs/heads/main"
    attestations:
    - predicateType: "https://slsa.dev/provenance/v1"
```

For: only signed + provenanced images.

## Best Practices

- L2 minimum for prod
- GitHub Attestation Action
- Cosign verify
- SBOM included
- Audit policy

## Common Mistakes

- Skip provenance
- Provenance without verification (theater)
- Mix L1 and L2 (consistency)
- No policy
