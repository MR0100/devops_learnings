## Blog Topics That Land

### Deep Dives (Most Valuable)
- "What I learned operating Prometheus at 5M samples/sec"
- "Migrating a 100-service estate from EC2 to EKS"
- "Building a FinOps program from scratch"
- "How we reduced p99 latency from 500ms to 80ms"

### Tutorials (Easier to Write)
- "End-to-end OIDC for AWS from GitHub Actions"
- "Setting up Cilium service mesh"
- "Migrating to Karpenter from Cluster Autoscaler"

### Opinion / Architecture (Engages Readers)
- "Why we picked Linkerd over Istio"
- "The hidden cost of multi-cloud"
- "Why your service mesh isn't paying off"

### Case Studies (Most Engagement)
- "We had a 3-hour outage. Here's what we learned."
- "How a single config change saved us $50K/month"

## Writing Strategy

### Frequency
- 1 post / quarter minimum
- More if comfortable
- Quality > quantity

### Outlets
- **Personal blog**: full control
- **Medium / Hashnode**: more reach
- **Company blog**: if employer allows
- **Substack**: newsletter format

### Authoring
1. Pick topic from real experience
2. Outline structure
3. Write draft (1 sitting if possible)
4. Edit ruthlessly
5. Get 1 reader feedback
6. Publish
7. Share on LinkedIn, Twitter

### What Makes a Post Land
- Specific (not generic advice)
- Numbers (concrete impact)
- Real stories (failures + recoveries)
- Code samples (working)
- Diagrams (architecture)
- Tight (no fluff)

## Open Source Contributions

### What Counts
- Real bug fixes in tools you use
- Documentation improvements (often easier path to start)
- New features in maintained projects
- Maintainer status of a small but useful tool

### What Doesn't (As Much)
- Typo fixes
- Stars on others' projects
- Forks without follow-up
- Toy projects with no users

### Strategy

#### Step 1: Find Project You Use
Pick a project you've operated. Read the codebase.

#### Step 2: Triage Issues
Find "good first issue" or related to your use case.

#### Step 3: Contribute
- Small initially
- Engage in PR review
- Build relationship with maintainers

#### Step 4: Sustain
- Multiple PRs over time
- Discussions in issues
- Eventually maybe maintainer status

### Examples of Strong OSS
- Prometheus exporter for your domain
- Terraform provider improvements
- K8s operator bug fixes
- New CLI for an underserved area
- Contributor to popular OSS (kubectl plugins)

## Building Personal Brand

### Topics to Own
Pick 1-2 areas to be associated with. Examples:
- "K8s networking" / "service mesh"
- "FinOps at scale"
- "Observability"
- "Multi-region architecture"

Become known for these. Easier to be deep on 1-2 than shallow on 10.

### Engagement
- Comment on others' posts (substantively)
- Reply to questions on Stack Overflow / Reddit
- Tweet thoughts (~3-5x week)
- Engage on Twitter / LinkedIn / Mastodon / Bluesky

### Don't
- Self-promote constantly (annoying)
- Claim credit for team work (look at "I" vs "we")
- Argue / be combative
- Recycle others' content without attribution
