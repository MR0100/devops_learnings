# L05/C07/T04 — Conventional Commits

## Learning Objectives

- Apply conventional commit format
- Enable automated versioning
- Generate changelogs

## Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

## Types

- `feat`: new feature
- `fix`: bug fix
- `docs`: documentation
- `style`: formatting; no code change
- `refactor`: code restructure; no behavior change
- `test`: tests
- `chore`: maintenance
- `build`: build system / dependencies
- `ci`: CI configuration
- `perf`: performance
- `revert`: reverting

## Examples

```
feat(auth): add OAuth login
fix(api): handle null in user response
docs: update README
refactor(db): extract connection pool
test(auth): cover edge case for expired tokens
chore: bump dependencies
```

## Scope

Optional. Identifies the area:
```
feat(auth): ...
fix(db): ...
docs(api): ...
```

## Breaking Changes

Two ways:

### `!` after type
```
feat(api)!: rename old endpoint
```

### `BREAKING CHANGE:` in footer
```
feat(api): rename foo to bar

BREAKING CHANGE: old endpoint /foo removed
```

Triggers MAJOR version bump.

## Semver Mapping

- `feat`: MINOR bump
- `fix`: PATCH bump
- `feat!` or `BREAKING CHANGE`: MAJOR bump
- `docs`/`style`/etc.: no version change

```
feat: 1.2.3 → 1.3.0
fix:  1.2.3 → 1.2.4
feat!: 1.2.3 → 2.0.0
```

## Automated Versioning Tools

### semantic-release (Node)
Reads commits since last release; bumps version automatically.

```yaml
# .releaserc.yml
plugins:
  - "@semantic-release/commit-analyzer"
  - "@semantic-release/release-notes-generator"
  - "@semantic-release/changelog"
  - "@semantic-release/git"
  - "@semantic-release/github"
```

In CI:
```yaml
- run: npx semantic-release
```

Bumps version + tags + creates GitHub release + updates changelog.

### release-please (Google)
Language-agnostic. Opens "release PRs" that bump version when merged.

```yaml
# GitHub Action
- uses: googleapis/release-please-action@v4
  with:
    release-type: simple
```

### goreleaser (Go)
Builds + releases Go projects on tag push.

### commitizen
Interactive commit message builder:
```bash
git cz       # opens wizard
```

## Changelog Generation

Conventional commits → human-readable changelog:

```markdown
## [1.3.0] - 2026-06-09

### Features
- auth: add OAuth login (#123)
- api: support pagination (#125)

### Bug Fixes
- db: connection leak under load (#127)
```

Auto-generated by semantic-release / release-please.

## Enforcement

### commit-msg Hook
```yaml
# .pre-commit-config.yaml
- repo: https://github.com/compilerla/conventional-pre-commit
  rev: v3.0.0
  hooks:
  - id: conventional-pre-commit
    stages: [commit-msg]
    args: [--strict]
```

Rejects non-conforming messages.

### commitlint
Node tool; widely used.

```bash
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

```javascript
// commitlint.config.js
module.exports = { extends: ['@commitlint/config-conventional'] };
```

In git hook:
```bash
# .husky/commit-msg
npx commitlint --edit "$1"
```

## CI Enforcement

```yaml
# GitHub Action
- uses: webiny/action-conventional-commits@v1.3.0
```

Fails PR if commits don't conform.

## When NOT Conventional

- Solo experimental projects
- Team prefers different format
- Don't need auto-versioning

For published libraries / serious projects: highly recommended.

## Squash Merge Tip

When squashing PR commits via GitHub:
- Edit final commit message to conform
- Or use PR title (often conforms naturally)

## Tools Around

- commitizen: interactive prompts
- standard-version: bumps + changelogs (predecessor of semantic-release)
- husky: install git hooks via npm scripts

## Common Mistakes

- Inconsistent scopes
- Vague subjects ("update stuff")
- Forgetting breaking change marker
- Multiple changes in one commit (bad)

## Style Tips

- Subject < 50 chars
- Subject in imperative mood ("add" not "added")
- Lowercase first letter (convention varies)
- No period at end of subject
- Wrap body at 72 chars

## Real Example

```
feat(payments): support partial refunds

Adds PATCH /refunds endpoint that accepts amount.
Validates amount against original charge.
Updates DB schema for partial refund tracking.

Closes #234
```

## Best Practices

- Pick a fixed, documented scope vocabulary (e.g. package or service names) so scopes are consistent and changelogs group cleanly.
- Mark breaking changes unambiguously with either `!` after the type/scope (`feat(api)!:`) or a `BREAKING CHANGE:` footer — and ideally both for clarity.
- Keep one logical change per commit so each maps to exactly one semver-relevant entry; squash WIP noise before merge.
- Enforce the format with two layers: a fast local `commit-msg` hook (commitlint / conventional-pre-commit) and a CI check as the real gate.
- For squash-merge workflows, lint the PR *title* (which becomes the squash commit) rather than every intermediate commit, so contributors aren't blocked on messy history.
- Wire the convention into automated release tooling (semantic-release / release-please) so version bumps, tags, and changelogs are derived, not hand-maintained.

## Quick Refs

```text
# Format
<type>(<optional scope>): <subject>     # <= 50 chars, imperative, no period

<optional body, wrap at 72>

<optional footer>
BREAKING CHANGE: <description>          # → MAJOR bump
Closes #123
```

| Type | Meaning | Semver |
|------|---------|--------|
| `fix:` | bug fix | PATCH |
| `feat:` | new feature | MINOR |
| `feat!:` / `BREAKING CHANGE:` | incompatible change | MAJOR |
| `docs/style/refactor/test/chore/ci` | non-shipping | none |

```yaml
# Enforce with commitlint (commit-msg hook) — .pre-commit-config.yaml
- repo: https://github.com/compilerla/conventional-pre-commit
  rev: v3.4.0
  hooks: [ { id: conventional-pre-commit, stages: [commit-msg] } ]
```

```bash
pre-commit install --hook-type commit-msg   # wire the commit-msg check
npx commitlint --from origin/main --to HEAD  # CI: lint a range of commits
npx semantic-release                          # derive version, tag, changelog
```

## Interview Prep

**Mid**: "What is Conventional Commits and what does it buy you?"
A structured commit format — `<type>(<scope>): <subject>` — that makes history machine-readable. The payoff: tools derive the next semver bump and generate changelogs automatically from commit history, with no manual version files.

**Senior**: "How do commit types map to semver?"
`fix:` → PATCH, `feat:` → MINOR, and either `feat!:` or a `BREAKING CHANGE:` footer → MAJOR. Non-shipping types (`docs`, `style`, `chore`, `ci`, `test`) produce no release. This mapping is what lets semantic-release / release-please bump and tag without human input.

**Staff**: "How do you enforce it across an org without slowing teams down?"
Two layers: a local `commit-msg` hook (commitlint / conventional-pre-commit) for fast feedback, plus a CI check on the PR as the real gate so it can't be bypassed. For squash-merge workflows, lint the PR *title* (which becomes the squash commit) rather than every intermediate commit, so contributors aren't blocked on messy WIP history.
