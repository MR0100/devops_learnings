## SLSA

Supply-chain Levels for Software Artifacts:
- Originated as a Google initiative, now an OpenSSF project
- Framework for build/supply-chain integrity
- Levels of assurance against tampering

The current spec is **SLSA v1.0 (2023)**. It is organized into **tracks**; the
mature one is the **Build track** with levels **L0–L3**. The legacy v0.1 model
(see historical note below) is no longer current.

## Build Track Levels (v1.0)

### L0
- No guarantees
- The default for most software

### L1
- Provenance exists (build process documented, provenance generated)
- Not necessarily signed or resistant to tampering

### L2
- Hosted build platform generates **signed** provenance
- Provenance is authenticated and tamper-evident

### L3
- Hardened, isolated build platform
- **Non-falsifiable** provenance (the builder, not the build, signs)
- Ephemeral, isolated build environments

For: progressive maturity. **There is no L4 in v1.0** — the old "two-person
review / hermetic builds" requirements were removed from the Build track. Review
controls now live in a separate, still-developing **Source track** rather than
being a build level.

### Historical note (v0.1)

The original SLSA v0.1 draft defined four build levels (L1–L4), where L4 added
two-person review and hermetic, reproducible builds. v1.0 retired L4, renumbered
to L0–L3, and split source-side controls into their own track.

## Level 2

Provenance:
- Signed
- Generated by build platform
- Not tampered

Tools: GitHub Actions, GitLab CI.

## Level 3

Build isolation:
- Ephemeral environment
- No long-lived secrets
- No interference

For: stronger.

## Source Track (review controls)

Two-person review and protected-branch controls moved here in v1.0. The Source
track governs how revisions enter the source repo (e.g. enforced review,
retained history) and is independent of the Build track levels above.

Hermetic and reproducible builds are still valuable goals (see T03), but in v1.0
they are best practices rather than a numbered build level.

## Threat Models

### Source
Attacker commits malicious code.
SLSA: signed commits, two-person review.

### Build
Attacker compromises CI.
SLSA: isolated builders, attestation.

### Dependency
Attacker compromises dep.
SLSA: SBOM + verify.

### Distribution
Attacker swaps artifact.
SLSA: signed + provenance verified.

## Interview Prep

**Junior**: "What is SLSA?" — A supply-chain integrity framework (now an OpenSSF project) that defines graded levels of build provenance so you can prove how an artifact was built and that it wasn't tampered with.

**Mid**: "What are the SLSA v1.0 build levels?" — L0 (no guarantees), L1 (provenance exists), L2 (signed provenance from a hosted build platform), L3 (hardened, isolated builder producing non-falsifiable provenance); there is no L4 in v1.0.

**Senior**: "What changed from v0.1 to v1.0?" — v0.1's L4 (two-person review + hermetic builds) was removed, levels renumbered to L0–L3, and source-side review controls were split into a separate Source track, so the Build track now measures only build integrity.

**Staff**: "How do you roll out SLSA across an org and tie it to compliance?" — Target L2 as the prod baseline via GitHub/GitLab signed provenance, push critical pipelines to L3 with ephemeral isolated builders, then enforce verification at admission (Cosign/policy) so unprovenanced images can't deploy — which also satisfies SBOM/attestation mandates like US EO 14028 and the EU CRA.
